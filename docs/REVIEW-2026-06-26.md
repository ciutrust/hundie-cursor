Review only — no code changes from this document. Fixes are written as follow-up prompts to run separately.

## How this review was done

This is the review the operator requested (the repo's own `PROMPT-2-codebase-review.md`).

**Methodology:** Read the six intent docs and every anchored file first-hand (suggestion engine, `category-expense.ts`, `reclassify.ts`, the migrations, all five issuer parsers, the import pipeline, the reporting queries, middleware, the Supabase clients). Then ran a multi-agent pass (11 agents) to cover breadth (app/components, the other queries, the ~65-script inventory) **and to adversarially verify every Critical/High/Medium claim against the real code.** Headline items were re-checked manually. One consequence worth stating up front: the automated pass raised a **Critical** ("re-import overwrites confirmed classifications") and a dependent **High** — and **both are false positives.** Disproved by the PostgREST `ignoreDuplicates` semantics (see Correctness §C12).

**Live DB:** Could **not** live-verify the deployed database. The connected Supabase tooling only exposes a *different, newer multi-tenant Hundie build* (`frqhmgrajfcmqthmeowi`), not this repo's project (`ihciuqpiavxhbulfkwod`, per `README.md:32`). RLS is verified **in the migration files** (and is correct) but production application of those migrations must be confirmed by you with the curl in `SUPABASE.md`.

Reviewer stance: skeptic, not cheerleader. Every issue cites `file:line`, a severity, why it matters **for a financial ledger**, and a concrete fix. Opinions are labeled as opinions. Real strengths are called out so the picture is honest.

**One-paragraph verdict.** Hundie is a genuinely well-built single-operator tool — the ledger-first discipline, the DB-enforced audit trail, the careful per-issuer sign normalization, and the byte-exact non-expense model are all things most teams get wrong and this one gets right. The previous review's headline (anon-readable RLS) has been correctly closed in a migration. What's left is a tier of **silent-correctness** risks specific to a books-of-record: a date rule that may be mis-booking an entire card's 2026 history, refunds dropped at import, a CSV export that doesn't reconcile with the on-screen totals, and a dedupe hash that can drop real charges. None corrupts stored data on its own, but each quietly produces a wrong number that flows to your CPA. Fix those, add ~8 unit tests on the pure functions, and the rest is upside.

## 2. Correctness risks that would corrupt the books

This is the heart of the review. Stored ledger data is generally safe (audit trail, dedupe, byte-exact non-expense matching all verified). The risks below are about **wrong numbers reaching the CPA**.

### Strengths (verified first-hand — don't lose these)
- **Non-expense exclusion is byte-exact.** Every entry in `NON_EXPENSE_CATEGORY_PATHS` (`lib/category-expense.ts:2-15`) — including the `→` (U+2192) and `—` (U+2014) glyphs — matches the seeded `full_path` in `20260626120000`, `20260627120000`, `20260629120000` exactly. This was the single highest books-corruption risk in scope and it's *correct*. (Fragile coupling, though — see C13.)
- **Sign convention is consistent and careful.** All five parsers normalize their issuer's quirk to "stored positive = charge" (WF/Chase keep `rawAmount < 0` then `abs`; Amex/CapOne/Citi keep debits). Both card-import paths (`import-cards.mjs` direct and `generate-card-import-sql.mjs`) produce **identical** rows — same sign, same hash, same dedupe — so they cannot double-insert or mis-sign.
- **Audit trail is tamper-resistant.** `classification_history` is written by a `security definer` trigger (`20260625160000:21-56`) on real entity/category changes only, and has a SELECT-only policy — no client can forge or edit it.
- **Re-import is safe.** Dedupe upsert `ignoreDuplicates` + classifications created only for newly-inserted rows means re-running an import never overwrites a human's confirmation (this is also why C12 is a *false* alarm).
- **Rental income / dormant entities handled correctly by construction.** `isExpenseAmount` requires `amount > 0` so negative rental income drops out automatically; reports filter `is_classifiable` so the dormant LLCs and trusts can't contribute to totals (`reports.ts:32`).

## 3. The suggestion engine — design critique

The engine is deterministic: token extraction → match QB-training (GBSL only) + confirmed ledger history + accept/reject events + amount-bucket re-rank → blend with hand-tuned weights → top 3 with confidence. This is the **right architecture** for this problem and it gets better the more it's used. My critique is about brittleness, cold-start, and legibility — not a call to replace it.

### Where it's brittle (file:line)
- **Hand-tuned magic numbers with no eval.** `blend-ranking.ts`: ledger `1.5×recencyWeight` (76), accept `2.5×recency` (89), reject `−0.5×recency` (97), `AMOUNT_EXACT_WEIGHT=6` / `AMOUNT_NEAREST_WEIGHT=4` (27-28); recency is a `3/2/1` step (18-25); `MIN_AMOUNT_BUCKET_COUNT=2` (amount-aware:1). These are reasonable guesses, but nobody can tell whether changing `2.5`→`3.0` helps or hurts because **there is no eval set.** You already log `suggestion_events` — the data to measure this exists and is unused.
- **The "count" on the chip is a fabricated score, not a count.** `displayCount = Math.max(1, Math.round(entry.score))` (`blend-ranking.ts:136`) is rendered as `· N×` (`category-suggestion-chips.tsx:141-143`). One exact-amount match (score 6×2=12) renders "12×" implying twelve sightings. The legacy non-blended paths (`category-suggestions.ts:266,302`) emit a *true* count, so the same `×` means two different things. For a human-in-the-loop tool, presenting a tuning score as evidence-of-frequency quietly inflates misplaced confidence. **(Medium — see S0_3.)**
- **Two-step source attribution is unreadable.** The 7-branch nested ternary picking `primarySource` (`blend-ranking.ts:120-134`) is then *re-overridden* at 153-156. Nobody but the author can predict which badge a suggestion gets.
- **Curated word lists are manual and load-bearing.** `STOP_WORDS`, `TAIL_STOP_WORDS`, `GENERIC_SINGLE_WORDS` (`category-suggestions.ts:30-69`). A vendor whose only distinctive token is on a list (e.g. a real merchant literally named "Square") collapses to no useful token and returns nothing.

### Cold-start: how much depends on having seen the vendor before
Trace `getCategorySuggestions` for a never-seen vendor with no QB-training match and no prior confirmation (the 3,000-row backlog case): `fetchQbTrainingRows` → `[]` (and only GBSL has QB training at all), `fetchLedgerRows` → `[]`, `fetchSuggestionEventRows` → `[]`, so `mergeWeightedSuggestions` returns `[]` and the chip shows "No matches — choose manually." **The engine contributes ~zero on cold-start, and that is structurally most of the backlog.** For the four non-GBSL entities there is *no* cold-start signal at all (no QB training). Concretely: the engine's value is almost entirely "the *second+* time it sees a vendor." That's not a flaw — it's the correct division of labor — but it means the backlog will not shrink from the deterministic engine alone.

### Maintainability for a non-author (Claudia / future dev)
Low. There is no single "score(suggestion) = …" you can read; the score is accreted across four loops with different weight shapes, then post-processed twice. "Why did it suggest that?" is not answerable from the code without simulating it. There is also no test pinning any of it, so a well-intentioned tweak can silently change rankings.

### Steelman: keep it exactly as-is
It's free, instant, fully explainable in principle (every contribution is a traceable integer/float), and **can never invent a category** — it only ranks categories that already exist in confirmed history/training. For the repeat-vendor weekly loop that is the steady state, an LLM would be slower, cost per-token, and risk drift for no benefit. The amount-band logic ($125 Gracie Barra = Software vs $850 = Franchise Fees) is structured knowledge an LLM does *worse* unaided. This engine is the right long-term spine.

### Steelman: augment it (the planned hybrid)
The engine's one gap is cold-start, and that's precisely the 3,000-row backlog that's also starving the learning loop of data. An LLM reads a messy never-seen description and makes a sensible first guess — exactly the cold case — written as **tagged "AI" suggestions, suggestions-only, human-confirmed** (the design in `PROMPT-3`). Every confirmed AI guess becomes training data the deterministic engine learns from. They compound; they don't compete.

### My recommendation (evaluated against the hybrid, not rip-and-replace)
1. **Keep the deterministic engine** as the ongoing learning loop. Do not replace it.
2. **Build the measurement harness *first*** — a tiny report of accept-rate by `suggestion_source`, entity, and confidence band from `suggestion_events` (the column and data already exist). This is ~half a day and it (a) turns "are the weights any good?" and "is AI better?" into data instead of opinion, and (b) is the only honest way to tune the magic numbers. **Do this before touching any weight.**
3. **Then add the LLM cold-start breaker** (`PROMPT-3`) for the backlog only. With the harness in place you'll know within a month whether to lean on it.
4. **Fix the chip "count" mislabel** regardless (cheap trust win).

Opinion, clearly labeled: the single highest-value change to the *engine* is not a weight — it's the eval harness. Without it, every tuning decision is a guess, and you can't answer the question you actually care about.

## 4. Architecture & maintainability

- **Server-action boundaries (mixed).** `reclassify` checks auth and surfaces `{ error }` correctly in the dialogs (`transaction-list.tsx:461-464` — a real strength). But `updateAccountSettings` and the two CSV exports don't check auth (S2-S3), and several actions return `{ error }` that isn't always shown: the **main report export button has no try/catch** so a thrown export fails silently (`report-export-button.tsx:16-31`) while its sibling *does* wrap it (`personal-card-business-export-button.tsx:27-29`) — the pattern is known, just omitted. There is **no `error.tsx`/`global-error.tsx`/toast** anywhere, so any *thrown* (vs returned) action has no graceful surface. **Fix:** add an `error.tsx`, wrap the export + reclassify awaits in try/catch, standardize on returned `{ error }` shown inline.
- **Query duplication (Low-Medium).** The paginate-by-1000 loop is copy-pasted **6×** (`reports.ts` ×2, `personal-card-business-report.ts` ×1 — with a *different* double-break condition — `review.ts` ×3), and `getCpaReviewCategoryIdSet` is re-fetched 5× in `review.ts`. Divergent copies are how an off-by-one creeps into a total. **Fix:** one `paginateAll<T>(buildQuery)` helper + memoize the CPA-review id set per request.
- **Types: generated format, statically committed, drifting risk (Medium).** `lib/types/database.ts` is in Supabase's generated shape (`__InternalSupabase`, `Row/Insert/Update/Relationships`) — *not* hand-written, good — but it's committed static with **no `gen:types` script**, so it can silently drift from migrations. It's in sync *now*. **Fix:** add `"gen:types": "supabase gen types typescript ... > lib/types/database.ts"` and run it in CI / a pre-commit check.
- **`scripts/` sprawl (Medium).** Canonical set = the **14** scripts referenced by `package.json` + their `lib/`. ## 5. Testing & verification

Today: **zero automated tests**, no test framework in `package.json`, a handful of `verify:*` scripts that hit the live DB or print parser output. The stack is ESM + Next 16 + React 19 + TS 6 → **Vitest** is the right choice (ESM-native, zero config for pure functions, no jsdom needed for the units below).

**Minimal setup:** `npm i -D vitest`; add `"test": "vitest run"`; one `vitest.config.ts` with `test.environment='node'`. Note `tsconfig` already `exclude`s `scripts/**/*.ts`, but the parsers are `.mjs` (importable directly) and the suggestion/expense logic is plain TS — all unit-testable as-is.

The 8 highest-value units (each catches a *books-corrupting* bug, all pure/trivially testable):

1. **`isOperatingExpense` / `NON_EXPENSE_CATEGORY_PATHS` byte-match** (`category-expense.ts`) — assert each non-expense path excludes, a real expense counts, **and** every Set entry exists as a seeded `full_path` (guards C13).
2. **`resolveEntitySlug`** (`entity-resolver.mjs`) — the 2025-06-30 / 2025-07-01 boundary, a 2026 date, overlap/ordering. **This is the C1 guard.**
3. **`buildTransactionHash`** (`import-hash.mjs`) — same inputs → same hash; and a regression test asserting two distinct charges that *should* differ do (drives the C3 fix).
4. **Each parser's sign/refund handling** (`*-csv-parser.mjs`) — a charge row → positive amount; a refund/payment row → dropped. Pins the convention so a format change fails loudly, not silently.
5. **`mergeWeightedSuggestions` + `recencyWeight`** (`blend-ranking.ts`) — pass a fixed `reference` date; assert ordering and that `count` is documented as a score (or, post-fix, a real count).
6. **`rankAmountAwareMatches`** (`amount-aware-ranking.ts`) — exact vs nearest, the `≥2` threshold, the no-ceiling behavior (C15).
7. **`extractSearchTokens` / `escapeIlikePattern`** (`category-suggestions.ts`) — tokenization of `GOOGLE *ADS…`, and that `%_,()` are stripped (S5 regression).
8. **`filterTransactions`** (`transaction-filters.ts`) — amount operators on `abs()`, review-backlog filter, search haystack.

Don't chase coverage %. These eight are the ones where a green test means "a wrong number can't silently ship."

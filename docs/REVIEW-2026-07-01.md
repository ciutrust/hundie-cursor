## How this review was done

Orchestrated multi-agent pass: **7 specialist finder agents** ran in parallel across four tracks (correctness ×2, security ×3, tooling, product), then every Critical/High security or correctness claim went to a **3-voter adversarial verification panel** (each voter assigned a different lens: factual correctness, real-world exploitability, intent/context) and every Medium got one skeptic — 41 agents total. A verifier's job was to *refute* the finding by tracing the real code path; anything unsubstantiated was dropped, and panel severity corrections are applied below. Live **Supabase security + performance advisors were fetched read-only from the actual production project** (`ihciuqpiavxhbulfkwod`) and cross-checked against the migrations — a first: previous reviews could not reach this project's live DB.

Findings were de-duplicated against the two prior audits ([REVIEW-2026-06-26.md](REVIEW-2026-06-26.md), [overnight tracker](overnight/2026-06-27/00-FINDINGS-TRACKER.md)). Items those reviews raised that are now **verified fixed** are not re-reported (middleware auth coverage, Quicksilver date rule, import-hash occurrence collisions, open redirect, CSV formula injection, `external_id` for Plaid modified events, eslint/typecheck tooling, `sync_from_date NOT NULL`). Where a finding below is a known-open item, it's tagged with its prior ID.

**One-paragraph verdict.** The foundation this team of reviewers expected to attack — crypto, Plaid route gating, RLS lockdown, sign conventions, dedup, the deterministic engine — largely held: AES-256-GCM token encryption verified end-to-end with no plaintext escape path, all five Plaid routes gate same-origin + auth + fail-closed MFA, the anon REST surface is effectively zero, and the ingestion pipeline's idempotency machinery is genuinely well built. The risk has moved to three places: (1) **one Supabase dashboard toggle** (public sign-ups) away from the entire "authenticated = owner" model collapsing — the login form can mint new users; (2) **the half-finished branch work**: the proposals commit path can silently overwrite newer manual classifications, and `/categories` sits outside the auth wall; (3) **silent-loss seams in ingestion**: unmapped Plaid accounts advance the cursor past dropped transactions, the CSV→Plaid cutover default can eat a date window, and the duplicate-cleanup script can delete genuine charges the import pipeline deliberately preserved. None of it is exploited or corrupting data today; all of it is the kind of quiet wrongness this app exists to prevent.

## 1. Executive summary — fix these first

1. **Close the self-registration hole (Security, High).** `signInWithOtp` has no `shouldCreateUser: false` ([components/auth/login-form.tsx:34](../components/auth/login-form.tsx)) and every RLS policy trusts any `authenticated` JWT. If "Allow new users to sign up" is enabled on the Supabase project (the default — nothing in the repo disables it), **a stranger can self-register and read/write the entire multi-entity ledger via PostgREST**. Check the dashboard toggle today; add `shouldCreateUser: false`; consider an email allowlist in policies. Five-minute fix, total-exposure downside.
2. **Make proposal commits respect newer human work (Correctness, High).** `commitApprovedProposals` upserts every approved proposal with no freshness check ([lib/actions/proposals.ts:125-140](../lib/actions/proposals.ts)) — a proposal approved days ago silently overwrites any classification (and wipes any notes) the operator made in `/review` since. Confirmed 3–0 by the adversarial panel. Skip rows whose classification is no longer `import`-authored, and never write `notes: null` over an existing note.
3. **Stop the Plaid sync from dropping transactions it can never get back (Correctness, High).** Transactions for unmapped accounts are `continue`d away while the cursor still advances ([lib/plaid/run-sync.ts:221-223, 277-285](../lib/plaid/run-sync.ts)) — Plaid's sync API never re-delivers them. Relatedly, `sync_from_date` defaults to link day, silently excluding the window between CSV end and Plaid link ([app/api/plaid/exchange/route.ts:43-57](../app/api/plaid/exchange/route.ts)). Don't persist the cursor when rows were unmappable; set/prefill the real cutover.
4. **Add CI and test the write layer (Tooling, High ×2).** 213 tests pass in 1.5 s offline — and gate nothing (no `.github/`). `lib/actions/` (1,626 lines, the only code that writes real classifications, including the service-role bulk committer reworked on this branch) has **zero tests**. A minimal workflow running `typecheck` + `lint` + `vitest` needs no secrets.
5. **Defuse the destructive-script footguns (Tooling/Correctness, Medium-High).** Bare `npm run cleanup:ledger-dupes` **deletes rows** (dry-run is the opt-in suffix, inverted from convention), and the script groups only by business key, ignoring `external_id` and occurrence hashes — it will delete genuine same-day/same-amount charges the importer deliberately preserved ([scripts/cleanup-ledger-duplicates.mjs:126-142](../scripts/cleanup-ledger-duplicates.mjs)). Flip bare names to dry-run; make the cleanup occurrence- and external_id-aware.

Honorable mentions: `/categories` (this branch) is missing from the auth matcher — three security agents found it independently; Plaid-removed transactions are stamped but no UI ever reads `plaid_removed_at`, so reversed charges count forever.
